# Codex 项目约定

这份文件定义本仓库的协作约束和提交要求。它描述的是当前真实运行形态，不是早期设计稿。

## 1. 变更分类

- 交易引擎、风控、执行、对账、私有流、数据库、部署、Web 控制面板、前端展示的变更，视为高影响变更。
- 高影响变更必须同时更新对应的 `docs/ops/` 中文变更记录，说明背景、改动、验证和线上影响。
- 仅文档改动时，也要保持与当前代码、systemd、`/etc/binance-trade`、`/opt/binance-trade`、`/var/lib/binance-trade` 的实际状态一致。

## 2. 当前架构边界

- 唯一源码仓库是 `/data0/binance-trade`。
- `/opt/binance-trade` 是 root 管理的发布副本，运行时不应手工改代码。
- 交易引擎和 Web 均由 `trader` 用户托管。
- `testnet` 与 `mainnet` 必须独立运行、独立数据库、独立日志、独立配置。
- Binance USD-M 私有 User Data Stream 是账户状态主通道，REST 只做启动基线、断线恢复和审计。

## 3. 编码要求

- 保持变更最小且贴合现有模块边界，不要引入与现有风格冲突的新抽象。
- 交易逻辑、风控逻辑和 Web 控制逻辑要明确分层，不能在前端或 API 层直接绕过引擎约束。
- 不允许硬编码密钥、令牌或运行时敏感响应。
- 对已存在的双环境、主网确认、命令队列、私有流分类逻辑，要优先沿用现有实现模式。

## 4. 验证要求

- 代码变更前先阅读相关源码、测试和 `docs/ops/` 记录，再动手修改。
- 交易、风控、对账、部署相关改动，提交前至少运行 `.venv/bin/python -m pytest`。
- Web 前端相关改动，至少运行 `web/frontend` 的构建检查。
- 如果改动涉及部署副本或 systemd，完成后要确认 `/opt/binance-trade` 同步一致，并检查服务状态。

## 5. 提交与发布

- 提交信息使用中文，简洁说明主题。
- 默认把代码改动和对应文档放在同一个提交里，方便回溯。
- 用户明确要求推送时，完成测试后推送到当前分支对应远端。
- 本项目 GitHub 远端使用 HTTPS：`https://github.com/iamxiaofu/binance-trade.git`。
- 推送默认直接使用本机已有的 Git HTTPS 凭据执行 `git push`；不要求安装或使用
  GitHub CLI `gh`，也不得因缺少 `gh` 停止提交或推送。
- 除非用户明确要求创建 Pull Request，否则不要套用依赖 `gh` 的 PR 发布流程。
- 当用户说“可以推送”或同义授权时，视为已授权将当前任务范围内、验证通过的改动
  提交到当前分支并通过 HTTPS 推送到其跟踪远端，无需再次询问。
- `.env`、数据库、日志、密钥、交易所原始敏感响应和其他运行时私密文件不允许提交。
